Swagger

// Introduction

Dans ce tutorial, Nous allons apprendre à créer un fichier YAML pour décrire une API qui effectue des opération de CRUD avec Swagger. CRUD signifié Creat, read, upadate et delete ( Créer, Lire, Mettre à jour et Supprimer ), ce sont les opérations les plus basiques qui peuvent être effectués sur une base de données. À la fin de ce tutoriel, nous aurons une solide compréhension de la création d'une API et de l'utilisation de YAML pour définir ses composants.

/ Comprendre les bases de YAML

YAML, qui signifie "YAML Ain't Markup Language", est un format de sérialisation de données lisible par l'homme. Il est souvent utilisé pour les fichiers de configuration, et c'est le format que Swagger utilise pour définir une API.

Avant de plonger dans la création d'un fichier YAML pour une API Swagger, il est important d'avoir une compréhension de base de la syntaxe YAML et des types de données.

Syntaxe YAML
  • YAML utilise l'indentation pour indiquer l'imbrication, similaire à Python.
  • YAML utilise les deux-points (:) pour séparer les clés des valeurs.
  • YAML utilise des tirets (-) pour indiquer des listes.
  • YAML utilise le symbole dièse (#) pour indiquer les commentaires.
Types de données YAML
  • Chaînes : entourées de guillemets ("") ou de guillemets simples ('')
  • Nombres : entiers ou nombres à virgule flottante
  • Booléens : vrai ou faux
  • Listes : indiquées par des tirets (-)
  • Dictionnaires : indiqués par deux-points (:)

Notez que l'indentation est importante dans YAML. Chaque niveau d'indentation indique un nouveau niveau d'imbrication.

  • Les fichiers YAML utilisent généralement l'extension de fichier .yml ou .yaml
  • Vous pouvez utiliser des outils en ligne tels que yamltojson.com pour convertir YAML en JSON et vice versa.

/ Comprendre l'importance de Swagger

Maintenant que nous comprenons les bases de YAML, parlons de l'importance de Swagger.

  • Swagger est un outil pour concevoir, créer et documenter des API REST
  • Il permet une collaboration facile entre les développeurs, les équipes d'assurance qualité et les parties prenantes
  • Il fournit une documentation claire et lisible par l'homme des points de terminaison de l'API, des exemples de requête/réponse, etc

En utilisant Swagger, nous pouvons nous assurer que notre API est bien conçue et facile à utiliser pour les autres développeurs. Il permet également une communication aisée entre les équipes et les parties prenantes.

  • Swagger dispose également d'un outil appelé Swagger Editor, qui permet de créer et de modifier facilement des fichiers YAML pour les API Swagger.
  • De nombreux langages de programmation populaires, tels que Node.js, disposent de bibliothèques qui s'intègrent à Swagger pour générer automatiquement de la documentation et d'autres fonctionnalités.

/ Création du fichier YAML pour notre API CRUD

Maintenant que nous comprenons les bases de YAML et l'importance de Swagger, nous pouvons commencer à créer le fichier YAML pour notre API CRUD.

Définissons d'abord la structure de base de notre fichier YAML :


swagger: '2.0'
info:
  description: 'un exemple d'API CRUD'
  version: '1.0.0'
  title: 'API CRUD'
host: 'localhost:3000'
basePath: '/'
schemes:
  - http
paths:
  # Nos opérations de crud serons là

Il s'agit de la structure de base d'un fichier Swagger YAML.

Il se compose d'un champ `swagger` qui définit la version de la spécification Swagger utilisée,

un champ `info` contenant des informations sur l'API telles que le titre, la version et la description

et un champ `paths` contenant tous les points de terminaison de l'API.

Dans le champ `paths`, nous pouvons définir nos opérations CRUD :


paths:
'/items':
  post:
    # Créer un item
  get:
    # Lire tous les items
'/items/{id}':
  put:
    # Mettre a jour un item
  delete:
    # Supprimer un item

Chaque point de terminaison est défini par son URL et la méthode HTTP utilisée pour y accéder.

Par exemple, le point de terminaison pour la création d'un nouvel élément est l'URL `/items` et la méthode HTTP est `post`.

Nous pouvons également spécifier les paramètres de demande et de réponse pour chaque opération.

  • Vous pouvez également utiliser les champs de `parameters` et de `responses` pour spécifier les paramètres de demande et de réponse pour chaque opération.

// Exercice

Faisons un fichier en YAML pour faire la documentation de notre futur API REST :

Le CRUD va s'effectuer sur des `items` avec 4 opérations :

  • Créer un nouvel élément
  • Obtenir une liste d'éléments
  • Mettre à jour un élément
  • Supprimer un élément

Voici un exemple possible pour cet exercice :


swagger: "2.0"
info:
  version: "1.0.0"
  title: "CRUD API"

paths:
  /items:
    post:
      summary: "Create a new item"
      description: "Create a new item with a unique ID and specified properties"
      parameters:
        - in: "body"
          name: "item"
          description: "Item object that needs to be added"
          required: true
          schema:
            $ref: "#/definitions/Item"
      responses:
        "201":
          description: "Item created successfully"
    get:
      summary: "Get a list of items"
      description: "Retrieve a list of all items"
      responses:
        "200":
          description: "List of items retrieved successfully"
          schema:
            type: "array"
            items:
              $ref: "#/definitions/Item"
  /items/{itemId}:
    get:
      summary: "Get an item by ID"
      description: "Retrieve an item by its unique ID"
      parameters:
        - in: "path"
          name: "itemId"
          description: "ID of the item to retrieve"
          required: true
          type: "integer"
      responses:
        "200":
          description: "Item retrieved successfully"
          schema:
            $ref: "#/definitions/Item"
    put:
      summary: "Update an item"
      description: "Update an existing item by its unique ID"
      parameters:
        - in: "path"
          name: "itemId"
          description: "ID of the item to update"
          required: true
          type: "integer"
        - in: "body"
          name: "item"
          description: "Item object that needs to be updated"
          required: true
          schema:
            $ref: "#/definitions/Item"
      responses:
        "200":
          description: "Item updated successfully"
    delete:
      summary: "Delete an item"
      description: "Delete an existing item by its unique ID"
      parameters:
        - in: "path"
          name: "itemId"
          description: "ID of the item to delete"
          required: true
          type: "integer"
      responses:
        "200":
          description: "Item deleted successfully"

definitions:
  Item:
    type: "object"
    properties:
      id:
        type: "integer"
      name:
        type: "string"
      description:
        type: "string"
